## 1. Problem framing

### 1.1 What the system needs

The frontend renderer (`WikiPageClient.tsx:91-93`) consumes
`<!--claim:N-->span<!--/claim:N-->` markers in page markdown and emits
`<span data-claim-id="N" id="claim-N">...</span>`, which `ClaimAnnotatedSpan`
(line 167) decorates with a trust-coloured dotted underline and a click-popover
showing `trust_level` + `evidence_count`. The rendering infrastructure is
**already shipped and verified**. It is starved by the absence of markers in
prose: a recent cleanup (`scripts/strip_markers_galaxy_evolution.py`) removed
the artificial D1–D10 debate markers from page 57's PV 3587 and the new content
pipeline has not been re-embedding them. Spot-check:

```text
SELECT COUNT(*) FROM wiki_pages WHERE content LIKE '%<!--claim:%';
→ 0 / 43
```

### 1.2 What makes this hard

A live alignment experiment on page 57 (41 claims vs. 492 prose sentences,
hybrid Jaccard-token + difflib SequenceMatcher score) returned:

| Metric                          | Value |
|---------------------------------|------:|
| Best fuzzy match score (any claim ↔ any sentence) | **0.35** |
| Mean best-score per claim       | 0.233 |
| Claims with best > 0.5          | **0 / 41** |
| Claims with best > 0.3          | 3 / 41 |
| Claims with best < 0.15 (no signal) | 3 / 41 |

**This is the load-bearing empirical finding of the entire design.** Claim text
and prose are **paraphrases of the same fact**, not the same string. They were
generated by different agents at different times: prose by AstroSage-70B /
Rakon during the v2 rewrite; claims by the claim-decomposer
(`scripts/decompose_claims.py`) seeded from earlier content. Token-overlap and
sequence-similarity heuristics are **structurally insufficient**. Any design
that depends on substring or fuzzy-string matching will fail silently and
attach claims to wrong sentences.

Example (cid=1471, the *best-scoring* claim in the page):

> **Claim:** "Galaxies located in dense clusters tend to experience environmental
> quenching earlier than those in less dense areas, known as field galaxies…"
>
> **Best-match sentence:** "JWST data indicate environmental quenching in
> protoclusters occurs 1.5 ± 0.3 Gyr earlier than in the field at z = 4–6."

These say roughly the same thing. No string algorithm will reliably pair them;
both human and embedding-based judgement will.

### 1.3 Localisation hint we *do* have

`claims.section` (varchar) is populated and exactly mirrors the page's H2
heading text. For page 57:

```text
Environmental Effects & Galaxy Clusters       5 claims
Galaxy Formation & Dark Matter Halos          3
High-Redshift Universe & Cosmic Star Formation History  5
Open Questions & Future Directions            8
Overview & Historical Context                 3
Quenching Mechanisms                          9
Star Formation & Gas Physics                  3
Structural Evolution                          5
```

Every section header in `wiki_pages.content` matches a `claims.section`
verbatim — confirmed by audit. This restricts the candidate prose for each
claim to one section's body (∼50–80 sentences instead of 492), a free **8×
search reduction**. Open Questions and headings without claims (Chemical
Enrichment, References) are skipped.

`claims.order_idx` exists but is **not** a positional pointer into prose; it is
the creation-order of the claim within the section. Useful as a weak prior
(claims with lower idx more likely to map to earlier sentences) but **not**
authoritative.

## 2. Algorithm — 3-stage cascade

The matching algorithm is a deterministic cascade. Each stage narrows the
search space and the final stage commits a span. Stages are designed so the
slowest model (Buddle, ~20 s/call) runs on the smallest possible candidate
set.

### Stage A — Section restriction (free, deterministic)

1. Parse `wiki_pages.content` with a markdown AST parser (`mistletoe` —
   already in `requirements.txt`; otherwise `markdown-it-py`).
2. Build a list of `(section_h2_title, body_text, body_ast_node)` tuples.
3. For each claim, look up its `section`; restrict search to the matching
   section's body. If no exact match, fall back to fuzzy-match (Jaccard ≥ 0.6)
   over section titles; if still no match, mark claim `section_unresolved` and
   skip — these are logged but never injected.

Page 57 audit: all 8 distinct claim sections resolve cleanly to H2 headings.

### Stage B — Embedding pre-rank (cheap, semantic)

For each section:

1. Sentence-split the body using a heading-aware splitter (handles lists,
   blockquotes, math; preserves paragraph and block boundaries). Sentences
   inside code fences, tables, and link text are **excluded** as injection
   sites (see §3.4).
2. Embed every candidate sentence with **`nomic-embed-text:v1.5`** (137 MB,
   768-dim, ~5 ms/sentence on CPU via Ollama). Cache embeddings keyed by
   sentence hash; invalidate on content rewrite.
3. Embed each claim's `text`. Compute cosine similarity sentence-by-sentence.
4. Retain top-K=3 candidates per claim with cos ≥ 0.45 (calibrated below).

**Why nomic-embed-text:** open, 137 MB, runs cheaply on the Mac Studio without
touching the active reasoner slot. **Atom-7B is intentionally *not* used at
this stage** — it is a classifier, not a similarity model; using its hidden
states for cosine matching wastes its calibration. We keep Atom-7B for the
*judge* role (Stage D).

**Cos threshold calibration:** On page 57's 41 claims × 492 sentences with
`nomic-embed-text`, expected top-1 cosine for true paraphrase pairs is ~0.55–
0.70 based on prior NebulaMind retrieval work (`research_idea_anchors`
build-out). Threshold 0.45 is **deliberately loose** — Stage C handles
disambiguation. The threshold's only job is to drop catastrophic non-matches
that would waste Buddle calls.

### Stage C — LLM alignment & span extraction (expensive, definitive)

For each `(claim, top-K candidate sentences)` tuple, **Buddle**
(`deepseek-r1:32b` on Mac Pro) receives a structured prompt:

```text
SYSTEM
You are aligning a wiki claim to its supporting sentence in a wiki page.
Given a claim and 3 candidate sentences from the same section, decide which
sentence (if any) is the prose that most directly asserts the claim. Then
identify the smallest contiguous span within that sentence that conveys the
claim. Output strict JSON.

USER
SECTION: {section_title}

CLAIM (id={claim_id}, trust_level={trust_level}):
{claim_text}

CANDIDATES:
[A] {sentence_A}
[B] {sentence_B}
[C] {sentence_C}

Return JSON: {
  "chosen": "A" | "B" | "C" | "none",
  "span_in_chosen": "<exact substring of the chosen sentence, ≤ full sentence>",
  "confidence": 0.0..1.0,
  "reason": "<one short clause>"
}
Rules:
- "span_in_chosen" must be a verbatim substring of the chosen candidate.
- Smaller span > larger span when both convey the claim equally.
- Choose "none" if no candidate asserts the claim (paraphrase too loose,
  topic mismatch). Confidence then captures certainty in the rejection.
- Do NOT span across markdown markers ([..](..), **..**, `..`).
```

Buddle is the right model here because:

- Reasoning-with-rationale is its sweet spot (per platoon roster v2).
- 32 B parameters carry enough domain knowledge to recognise paraphrase
  equivalence in astronomy without AstroSage-70B's RAM cost.
- ~120 calls (41 claims × 3 candidates ≈ but Buddle sees all 3 in one call;
  41 calls total for page 57) finishes in ~15 minutes on Mac Pro.

**Fallback rule:** if Buddle returns confidence < 0.6 *and* the claim contains
domain-specific tokens flagged by Atom-7B's astronomy tagger (e.g., named
papers, instrument-specific quantities), re-run via **AstroSage-70B** on Mac
Studio. AstroSage's astronomy calibration breaks ties Buddle can't on a small
minority of edge cases (estimated < 10 % of claims).

### Stage D — Independent judge (Atom-7B)

After Stage C, **Atom-7B** independently re-scores each `(claim_text,
candidate_span)` pair without seeing Buddle's chosen candidate. It returns:

```json
{
  "agreement_score": 0.0..1.0,   // how well span_in_chosen conveys the claim
  "trust_alignment": "ok" | "mismatch"  // is span tone consistent with claim's trust_level
}
```

Used as a **veto gate**, not a re-ranker: if `agreement_score` < 0.6, the
injection for that claim is dropped. Atom-7B was fine-tuned on astronomy
arXiv; for our claim-vs-prose task this is the same family of judgement it
already does well on. Cost: ~5 ms/call, effectively free.

### Stage cascade summary

```text
Stage A (Python, free)        ── 41 claims  → 8 section buckets
Stage B (nomic-embed-text)    ── ~5 ms × (claims × section_sents)
Stage C (Buddle 32B)          ── ~20 s × 41 calls (≈ 14 min/page)
Stage D (Atom-7B judge)       ── ~5 ms × 41 calls (~negligible)
──────────────────────────────────────────────────────────
Per-page total wall-clock:    ~15 min on idle hardware
Per-page total cost:          $0 (all local models)
```

## 3. Injection pipeline

### 3.1 Why this needs an AST, not regex

The body content is markdown with bold, italic, links, lists, blockquotes,
inline code, and math notation (`δρ/ρ ~ 10⁻⁵`, `n_s = 0.9649 ± 0.0042`). A
regex injection will break in any of these scenarios:

- Span happens to start inside `**bold**` and end outside → unmatched `**`.
- Span overlaps `[anchor text](url)` → broken link node.
- Span sits inside ```` ```code``` ```` → rendered as literal `<!--claim:N-->`.
- Span begins/ends mid-word (no whitespace boundary) → no visible underline.

The injector therefore operates on a parsed AST and only mutates **text-node
content** within paragraph / list-item / blockquote contexts. Injection sites
that would require splitting an inline span node (link, emphasis, code) are
**rejected with a re-search**: the LLM is asked to choose a smaller span that
fits inside a single text node, or to choose `"none"`. We do not split inline
markup.

### 3.2 Injection procedure (per page)

1. Build the AST.
2. Collect all `(claim_id, chosen_section, candidate_span)` tuples that
   passed Stage C+D.
3. For each tuple, locate the text node `T` inside the section's body whose
   content contains `candidate_span` as a substring (and only one such occurrence
   exists — see §3.5 for ambiguity handling).
4. Split `T` into `[prefix] + [<!--claim:N-->] + [span] + [<!--/claim:N-->] + [suffix]`
   as raw text. Markdown comment syntax `<!--...-->` is a passthrough in
   CommonMark and is preserved by `mistletoe.markdown_renderer`.
5. After all injections in a page are batched, re-serialise the AST.
6. Validate (§3.6) and persist.

Injections within a single page are batched and applied in **descending
section order, then descending sentence order** so character offsets stay
stable. Per claim, only one injection (the first eligible site).

### 3.3 Forbidden injection sites

The injector enforces a deny-list before considering a candidate span:

| Site                    | Reason                                          |
|-------------------------|-------------------------------------------------|
| Inside `H1/H2/H3`       | Headings already serve as anchors; double-annotation |
| Inside fenced code      | Renders literally; breaks code                  |
| Inside inline code      | Same                                            |
| Inside link text `[..]` | Splits the anchor; breaks link                  |
| Inside emphasis `**..**` | Splits emphasis run                            |
| Math `$..$` / `$$..$$`  | Breaks KaTeX parsing                            |
| List bullet marker (`- `, `1. `) | Disturbs list rendering                  |
| References section      | These are citation lists, not claim assertions  |
| Pre-existing claim wrap | Idempotency (no double-wrap)                    |

### 3.4 Sentence splitter quirks

A sentence-aware splitter that respects:

- Greek letters, scientific notation, decimal numerals, and `±` (so
  "n_s = 0.9649 ± 0.0042" is not split mid-equation).
- Citations: `(White & Rees 1978)` stays glued to its sentence.
- Abbreviations: `et al.`, `e.g.`, `i.e.`, `cf.`, `Fig.`, `Eq.`.
- Equation lines: standalone `$$ ... $$` blocks treated as code-fence
  equivalents (no claims injected).

Implementation: `pysbd` (Pragmatic Sentence Boundary Disambiguation) with a
custom astronomy abbreviation list. `pysbd` is in
`requirements.txt` for the existing `astro_sentence_split.py` utility — reuse.

### 3.5 Ambiguity handling

If the span returned by Stage C appears more than once in the section
(e.g., a short phrase like "AGN feedback"), Stage C's prompt is re-issued
with the instruction "return a span that occurs only once in the candidate
sentence." If still ambiguous after 2 retries → drop the claim's injection
(logged as `ambiguous_span`).

### 3.6 Post-injection validation (page-level)

Before committing the new `wiki_pages.content`:

1. **Markdown round-trip:** parse → serialise → parse again. AST structure
   must be identical to pre-injection AST modulo inserted HTML-comment nodes
   inside text content.
2. **No nested or unmatched markers:** regex assertion `<!--claim:(\d+)-->`
   count equals `<!--/claim:\1-->` count, and no two opening markers share
   an id.
3. **No marker spans block boundary:** for each `(open, close)` pair, both
   markers must live inside the same paragraph / list-item / blockquote AST
   node.
4. **Frontend smoke test:** run the same regex used by
   `WikiPageClient.tsx:91`; the number of resulting `<span data-claim-id>`
   tags must equal the number of injected claims for the page.
5. **Coverage floor:** matched_count / total_claims ≥ 0.60. If below, the
   page write is rejected and queued for manual review.

All validations log to a new table `claim_marker_runs`:

```sql
CREATE TABLE claim_marker_runs (
  id              SERIAL PRIMARY KEY,
  page_id         INT NOT NULL REFERENCES wiki_pages(id),
  page_version    INT,                        -- new PV after injection
  source_version  INT,                        -- PV at run start
  total_claims    INT,
  matched_claims  INT,
  rejected_low_confidence INT,
  rejected_no_section     INT,
  rejected_ambiguous_span INT,
  rejected_validation     INT,
  mean_confidence FLOAT,
  judge_agreement_pct FLOAT,
  coverage_pct    FLOAT,
  status          VARCHAR(32),                -- 'committed' | 'rolled_back' | 'partial'
  run_started_at  TIMESTAMP,
  run_finished_at TIMESTAMP,
  notes           TEXT
);
```

## 4. Platoon assignment

Per Papa's "Platoon Assignment in design docs for periodic/real-time jobs"
directive (memory: `feedback_platoon_assignment.md`), every step is owned by a
named model with capability + cost + speed justification.

| Stage | Step                              | Model                | Host       | Why                                                  | Cost      | Speed    |
|------:|-----------------------------------|----------------------|------------|------------------------------------------------------|-----------|----------|
| A.1   | Parse content, build section map  | Python (`mistletoe`) | Mac Studio | Deterministic; no model needed                       | $0        | <100 ms  |
| A.2   | Resolve `claims.section` ↔ H2     | Python               | Mac Studio | Exact string + Jaccard fallback                      | $0        | <50 ms   |
| B.1   | Sentence-split content            | `pysbd` (Python)     | Mac Studio | Domain-aware splitter; already in stack              | $0        | <500 ms  |
| B.2   | Embed sentences + claims          | `nomic-embed-text`   | Mac Studio | 137 MB; co-resident with Atom-7B; fast cosine        | $0        | ~5 ms/sent |
| B.3   | Top-K=3 cosine ranking            | Python (`numpy`)     | Mac Studio | Trivial                                              | $0        | <1 s     |
| C.1   | Align claim → span                | **Buddle (32B)**     | Mac Pro    | Reasoning-with-rationale specialty; cheap on Pro     | $0        | ~20 s/call |
| C.2   | Astronomy edge-case fallback      | **AstroSage-70B**    | Mac Studio | Domain calibration when claim names a paper/instrument | $0      | ~30 s/call |
| D.1   | Independent judge re-score         | **Atom-7B**          | Mac Studio | Astronomy classifier, free, always-on candidate      | $0        | ~5 ms/call |
| —     | Pipeline orchestration             | Tori                 | Mac Studio | Schedules Celery task, batches Buddle calls          | Cloud-paid | —       |
| —     | Algorithm tuning & post-mortem     | Kun                  | Mac Pro    | Threshold calibration, audit-report review           | Cloud-paid | —       |
| —     | Run scheduling                     | HwaO                 | Mac Studio | Avoid Rakon co-residency on Mac Pro                  | Cloud-paid | —       |

### 4.1 Hardware-scheduling constraints (RAM contention)

Buddle (~20 GB on Mac Pro) cannot run concurrently with Rakon (404 GB
exclusive). HwaO must ensure no Rakon jobs are scheduled while a marker-embed
pipeline run holds Mac Pro. Mac Studio side is unconstrained: nomic-embed-text
(0.15 GB) + Atom-7B (5 GB) + (optional) AstroSage-70B (42 GB) co-reside with
no thrash.

### 4.2 Why not just use one big model

A naïve design would prompt AstroSage-70B with all 41 claims and the entire
page in one shot and ask it to insert markers directly. Rejected because:

- Single-call output of ~84 kc with surgically-precise marker injections is
  beyond reliable LLM output discipline (markdown breakage rate >> 0).
- No independent judge → no false-match detection.
- No granular logging — when a marker is wrong, we can't tell *why*.
- Doesn't generalise: a page rewrite would force a full re-prompt instead of
  re-running only Stage C for changed sentences.

The cascade is built so each stage is independently auditable and the
expensive stage (C) sees the smallest possible input.

## 5. Freshness strategy

The pipeline must coexist with the **content rewrite cycle**. Pages get
rewritten by `run_coherence_v2.py` and the agent-loop coherence/renovation
passes; markers in old content cannot be assumed valid against new prose.

### 5.1 Marker storage: in-content (single source of truth)

Markers live in `wiki_pages.content`. There is no separate marker overlay
table. Rationale:

- The frontend regex already consumes `wiki_pages.content` directly. Adding
  an overlay layer means a new join in the page-fetch path — perf cost for
  no benefit since the embed pipeline is cheap.
- Markers travel with the content through `PageVersion` history — past
  versions remain renderable without resurrecting old overlays.

### 5.2 Re-embed-on-rewrite

Every content rewrite triggers a re-embed run. Wiring:

1. Add a `MARKER_REEMBED_REQUIRED` event emit at the tail of every step that
   updates `wiki_pages.content`:
   - `run_coherence_v2.py` (Opus / AstroSage coherence pass)
   - `app/agent_loop/auto_improvement.py` rewrite steps
   - Manual `update_page_content` via `app/routers/pages.py`
2. A Celery task `claim_marker_embed_page(page_id)` consumes the event:
   - Diffs old vs. new content. If unchanged, no-op.
   - Otherwise runs the full §2 cascade and writes a new PageVersion.
3. Embed cache (Stage B) keyed by sentence-hash means unchanged paragraphs
   skip re-embedding; only rewritten sections cost the LLM stage.

### 5.3 Pointer-rot is not a concern

Because markers are re-derived from current prose every time prose changes,
there is no "stale pointer." A claim that no longer maps to any sentence
(e.g., topic removed) simply doesn't get a marker on the new content and is
logged as `unmatched_after_rewrite`. Renovation pipeline (`설계_WikiRenovation_v1.md`)
already produces such claims as candidates for `claim_decay_candidates`.

### 5.4 Idempotency guarantee

The injector strips all existing `<!--claim:*-->` from input content before
running the cascade. Re-running on the same content twice produces the same
output. This makes the Celery task safely retryable.

## 7. Quality gates summary

| Gate | Stage | Action on failure | Logged as |
|------|-------|-------------------|-----------|
| Section unresolved | A | Drop claim from this run | `rejected_no_section` |
| Cosine top-1 < 0.45 | B | Drop claim | `rejected_low_similarity` |
| Stage C confidence < 0.7 | C | Drop claim | `rejected_low_confidence` |
| Stage C says "none" | C | Drop claim | `rejected_llm_reject` |
| Stage D agreement < 0.6 | D | Drop claim | `rejected_judge_veto` |
| Span not unique in section | injector | Retry with smaller span; if 2 retries fail, drop | `rejected_ambiguous_span` |
| Span overlaps forbidden site (§3.3) | injector | Retry with smaller span; if fail, drop | `rejected_unsafe_span` |
| Markdown round-trip mismatch | validation | Roll back entire page run | `rejected_markdown_break` |
| Page coverage < 60 % | validation | Roll back entire page run, manual review queue | `rejected_low_coverage` |

These deliberately stack on the side of caution. A claim that *should* have
been matched but was dropped is recoverable (log → re-tune → re-run). A
claim incorrectly attached to the wrong sentence is **invisible to the
audit** once committed — users see an underline and trust it. So thresholds
err high.

### 7.1 Anti-pattern guards

Lessons from prior systems (memory: `feedback_multivote_inflation_antipattern.md`,
`feedback_judge_saturation_design.md`):

- **No saturating rubrics.** Buddle's confidence is a float in [0,1], not an
  integer score. Threshold (0.7) is a hard gate, not part of an additive total.
- **Judge is a veto, not a re-ranker.** Stage D cannot promote a claim Buddle
  dropped, only block one Buddle accepted. Prevents Atom-7B's high astronomy
  recall from masking Buddle's prose-alignment failures.
- **One claim → one marker.** Per-claim deduplication is enforced before the
  injector touches the AST. No parallel-rerun row inflation.
